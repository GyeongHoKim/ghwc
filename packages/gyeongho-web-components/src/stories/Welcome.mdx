# My Boss Hates React, Angular, Vue

When I started my current job, I was told that my boss does not like React.
There are many reasons for this, but the main one is that he believes that modern FE frameworks are too complex for our project.
Of course, I disagree with him, but I understand his point of view.
As a result, I am now working with Lit and Web Components during my working hours.

# Design Systems

After a few months, I realized that there are no design systems in my company.
However, I was the only front-end developer & designer in my company and there was no time for me to create a design system.
At that time, I used Shoelace and Carbon design systems.

They are pretty good for our team, but I always wanted to create our own design system.
This project is practice for creating design systems for my company.

For now, I plan to create button, modal, input, and table components.
The practice period is one month.
Based on the practice experience, exactly one month later, the ultimate goal is to go to work, quickly build a design system, and actually use it.

# Inconveniences experienced while using open-source design system libraries at work

- Almost all of them can only be used in React. If you create a common library with React components, what about teams that don't use React?
- They are very heavy. We want something lightweight.
- Each component should be packaged separately.
- The components I want are missing. My requirements are as follows:
- Refer to the requirements document for each component.
- Styles and icons vary depending on the client, and there are limitations with themes.
- The more detailed the library's parts were, the more convenient it was. But there aren't many such libraries.
- Of course, themes are also necessary.
- It should be headless and allow injecting CSS from the outside.
- It's possible with React, but on the web component side, no alternatives come to mind other than themes.
- Research is needed.
- There are no components related to layout.
- All spacing, padding, margins, etc. should be the same, but they are the same only within each component, and the spacing between components is up to the developer.

# 디자인 시스템

몇 달 후, 저는 우리 회사에 디자인 시스템이 없다는 것을 깨달았습니다.
그러나 저는 회사에서 유일한 프론트엔드 개발자이자 디자이너였고, 디자인 시스템을 만들 시간이 없었습니다.
그 당시에는 Shoelace와 Carbon 디자인 시스템을 사용했습니다.

우리 팀에게는 꽤 좋은 선택이었지만, 저는 항상 우리만의 디자인 시스템을 만들고 싶었습니다.
이 프로젝트는 우리 회사를 위한 디자인 시스템 제작 연습입니다.

당장은 버튼, 모달, 인풋, 테이블 컴포넌트를 만들어보려고 합니다.
연습 기간은 한 달입니다.
연습 경험을 토대로 정확히 한 달 뒤, 회사에 출근해서 디자인 시스템을 뚝딱 구축하고 실제로 사용하는 것이 최종 목표입니다.

# 회사에서 오픈소스 디자인 시스템 라이브러리를 사용하면서 느낀 불편한 점들

- 거의 대부분이 리액트에서만 사용할 수 있습니다. 공통 라이브러리를 리액트 컴포넌트로 만들면 리액트를 사용하지 않는 팀은 어떻게 해야 할까요?
- 엄청 무겁습니다. 우리는 가벼운 것을 원합니다.
- 각 컴포넌트는 별도로 패키징되어야 합니다.
- 제가 원하는 컴포넌트가 없습니다. 제 요구사항은 다음과 같습니다:
- 각 컴포넌트에 대한 요구사항 문서를 참조하세요.
- 고객사에 따라 스타일과 아이콘이 달라지는데 테마로는 한계가 있습니다.
- 라이브러리의 부분이 자세할수록 편리했습니다. 하지만 그런 라이브러리는 많지 않습니다.
- 물론 테마도 필요합니다.
- Headless여야 하며 외부에서 CSS를 주입할 수 있어야 합니다.
- 리액트에서는 가능하지만, 웹 컴포넌트 쪽에서는 테마 외에 다른 대안이 떠오르지 않습니다.
- 연구가 필요합니다.
- 레이아웃과 관련된 컴포넌트가 없습니다.
- 모든 간격, 패딩, 여백 등이 동일해야 하는데, 각 컴포넌트 내에서만 동일하고 컴포넌트 간의 간격은 개발자 마음대로입니다.
